## 一、整体架构

Claude Code 的多 Agent 体系由四个正交维度组合而成：

```
执行位置          × 生命周期           × 隔离级别          × 编排模式
─────────────────────────────────────────────────────────────
进程内 (in-process)   同步 (sync)         无隔离             普通模式
后台任务 (background) 异步 (async)        worktree 隔离      协调器模式
远端 CCR (remote)     长驻 (long-running) 远程容器           Fork 实验模式
Pane-based (tmux/iTerm2)  Teammate 编队                      Swarm 模式
```

## 二、AgentTool — 统一入口

文件：`tools/AgentTool/AgentTool.tsx`

AgentTool 是所有子 Agent 的唯一入口，内部按条件分发到四条路径。

### 2.1 输入 Schema

```
AgentToolInput
├── description         # 3-5 词任务描述（必填）
├── prompt              # 具体任务内容（必填）
├── subagent_type       # 指定 Agent 类型（omit = 自动路由）
├── model               # 可选模型覆盖 (sonnet | opus | haiku)
├── run_in_background   # 是否后台运行（true = 立即返回 async_launched）
│
│── [KAIROS gate] ──────────────────────────────────────────
├── name                # 赋名 → 可被 SendMessage({to: name}) 寻址
├── team_name           # 目标团队名
├── mode                # Teammate 权限模式（如 "plan"）
├── cwd                 # 绝对路径（互斥于 worktree 隔离）
│── [always] ───────────────────────────────────────────────
└── isolation           # worktree | remote（ant-only 支持 remote）
```

### 2.2 输出类型

| status | 触发条件 |
|--------|---------|
| `completed` | 同步子 Agent 完成 |
| `async_launched` | 后台 Agent 已注册，返回 agentId + outputFile |
| `teammate_spawned` | Swarm 编队成员启动（含 tmux_pane_id 等 UI 字段） |
| `remote_launched` | 远端 CCR 会话已创建，返回 sessionUrl |

### 2.3 分发逻辑（call() 内部路由）

```
call() 入口
  │
  ├─ teamName && name  →  spawnTeammate()        [Swarm 路径]
  │
  ├─ isolation === 'remote'  →  teleportToRemote()  [CCR 远端路径]
  │
  ├─ isForkSubagentEnabled() && subagent_type omitted
  │     →  FORK_AGENT (父会话上下文继承)            [Fork 实验路径]
  │
  └─ otherwise  →  runAgent(selectedAgent)         [普通子 Agent 路径]
        │
        ├─ shouldRunAsync  →  registerAsyncAgent() + background
        └─ !shouldRunAsync →  await runAgent() (同步，阻塞当前轮次)
```

### 2.4 强制异步触发条件

以下任意条件满足时，子 Agent 自动异步化（`shouldRunAsync = true`）：

```
run_in_background === true
selectedAgent.background === true
isCoordinatorMode()
isForkSubagentEnabled()            (FORK_SUBAGENT gate)
appState.kairosEnabled             (Assistant 模式 / KAIROS gate)
proactiveModule.isProactiveActive()
```

额外保险：同步 Agent 超过 120s 自动切后台（`AUTO_BACKGROUND_MS = 120_000`），
由 GrowthBook 或 `CLAUDE_AUTO_BACKGROUND_TASKS` 环境变量启用。

## 三、AgentDefinition — Agent 注册表

文件：`tools/AgentTool/loadAgentsDir.ts`

每个 Agent 类型在注册表中注册一个 `AgentDefinition`：

```
AgentDefinition
├── agentType           # 唯一标识符（e.g. "general-purpose"）
├── description         # 显示给 LLM 的用途描述
├── prompt / getSystemPrompt()  # 子 Agent 系统提示构建函数
├── tools               # 允许工具白名单（'*' = 继承父工具集）
├── disallowedTools     # 工具黑名单
├── model               # 专属模型（可被调用方 override）
├── permissionMode      # 默认权限模式
├── isolation           # 默认隔离模式（worktree / remote）
├── background          # true = 总是异步
├── requiredMcpServers  # 启动前必须存在的 MCP 服务器（glob 匹配）
├── mcpServers          # Agent 专属 MCP 服务器（inline 定义或名称引用）
├── color               # UI 颜色标识
├── memory              # 记忆范围（auto / project / team）
├── hooks               # Agent 专属 Hook 脚本
└── source              # built-in | agents-dir | plugin
```

**加载来源（优先级由高到低）：**

1. `getBuiltInAgents()` — 代码内置（`tools/AgentTool/built-in/`）
2. `loadMarkdownFilesForSubdir('agents')` — 磁盘 `.claude/agents/*.md`（前置 matter 解析）
3. `loadPluginAgents()` — 插件扩展

**过滤管线：**
```
filterAgentsByMcpRequirements()   # 筛掉 MCP 服务器未就绪的 Agent
filterDeniedAgents()              # 应用 alwaysDeny 权限规则
```

## 四、三种本地执行模式

### 4.1 普通同步子 Agent（Subagent）

**触发条件**：`shouldRunAsync = false`，常见于默认模式下的工具委托。

**执行流程**：
```
AgentTool.call()
  │
  ├── 构建子 Agent SystemPrompt（selectedAgent.getSystemPrompt()）
  ├── 组装工具集（assembleToolPool，独立于父工具集）
  ├── 构建 promptMessages：[UserMessage(prompt)]
  ├── runAgent()  ← 进程内同步循环（直到 end_turn 或 abort）
  └── 将 task-notification XML 注入父会话消息队列
```

**资源隔离**：
- 独立 `abortController`
- 克隆文件状态缓存 `cloneFileStateCache()`
- 独立 `messages[]`（不共享父会话历史）

### 4.2 后台异步子 Agent（LocalAgentTask）

文件：`tasks/LocalAgentTask/LocalAgentTask.tsx`

**触发条件**：`shouldRunAsync = true`，注册后立即返回。

```
registerAsyncAgent()
  ├── 生成 taskId (UUID)
  ├── AppState.tasks.set(taskId, LocalAgentTaskState)
  └── 启动 runAgent() 为 Promise（不 await）

AgentTool.call() 返回：
  { status: 'async_launched', agentId, outputFile, ... }
```

**进度追踪（ProgressTracker）**：
```
ProgressTracker
├── latestInputTokens    # 最新一轮 input tokens（不累加，避免双计）
├── cumulativeOutputTokens  # 输出 tokens 累计
├── currentTool          # 当前正在执行的工具
├── toolActivities[]     # 工具活动历史（最近 N 条）
└── getProgressUpdate()  # 生成人类可读进度摘要
```

**完成通知（task-notification XML）**：
```xml
<task-notification>
  <task-id>xxx</task-id>
  <tool-use-id>xxx</tool-use-id>
  <task-type>local_agent</task-type>
  <output-file>/path/to/output</output-file>
  <status>completed</status>
  <summary>Task finished with result: ...</summary>
</task-notification>
```
由 `enqueuePendingNotification()` 注入主 Agent 的消息队列，以 user 角色呈现。

### 4.3 Worktree 隔离

**触发条件**：`isolation === 'worktree'`（参数或 AgentDefinition 默认值）。

```
createAgentWorktree(slug)
  ├── git worktree add .agent-worktrees/{slug} -b agent/{slug}/{head}
  ├── 返回 { worktreePath, worktreeBranch, headCommit, gitRoot }
  └── 设置 cwd override → runWithCwdOverride(worktreePath, ...)

cleanupWorktreeIfNeeded()（完成后）
  ├── hasWorktreeChanges(worktreePath, headCommit)
  │     true  → 保留 worktree，返回 { worktreePath, worktreeBranch }
  │     false → removeAgentWorktree() (git worktree remove)
  └── hook-based worktrees 总是保留（无法检测 VCS 变更）
```

## 五、远端 CCR（RemoteAgentTask）

文件：`tasks/RemoteAgentTask/RemoteAgentTask.tsx`

**触发条件**：`isolation === 'remote'`（仅 ant 构建可用）。

### 5.1 前置检查

```
checkRemoteAgentEligibility()
  ├── 已登录 claude.ai 账号（非 Console）
  ├── 有远端环境（cloud environment）
  ├── 当前目录是 git repo
  ├── 有 GitHub remote
  ├── GitHub App 已安装
  └── 组织策略未阻止
```

### 5.2 任务类型

| RemoteTaskType | 描述 |
|----------------|------|
| `remote-agent` | 标准远端子 Agent |
| `ultraplan` | 远端计划生成（ExitPlanMode 等待浏览器审批） |
| `ultrareview` | 远端代码审查（含 finding/verifying/synthesizing 阶段） |
| `autofix-pr` | 自动修复 PR |
| `background-pr` | 后台 PR 创建 |

### 5.3 状态追踪

```
RemoteAgentTaskState
├── sessionId           # 远端会话 ID
├── pollStartedAt       # 轮询开始时间（用于 resume 后超时校准）
├── ultraplanPhase      # needs_input | plan_ready（ExitPlanMode 触发）
├── reviewProgress      # { stage, bugsFound, bugsVerified, bugsRefuted }
├── log: SDKMessage[]   # 远端会话事件流
└── todoList            # 远端 TodoWrite 状态同步
```

远端轮询由 `pollRemoteSessionEvents()` 驱动，
完成后通过 `enqueueRemoteNotification()` 注入主会话 task-notification。

## 六、Fork 子 Agent（实验特性）

文件：`tools/AgentTool/forkSubagent.ts`

**触发条件**：`FORK_SUBAGENT` feature gate 开启，且 `subagent_type` 未指定。

### 6.1 核心设计

Fork 子 Agent 的核心目标是**最大化 Prompt Cache 命中率**：

```
FORK_AGENT 定义
├── agentType: 'fork'
├── tools: ['*']                   # 继承父工具集（bit-for-bit 相同）
├── permissionMode: 'bubble'       # 权限决策冒泡到父 Agent
└── model: 'inherit'               # 继承父 Agent 模型

Fork 路径特殊处理（AgentTool.call()）
├── systemPrompt → 直接使用父 Agent 的 renderedSystemPrompt
│     （避免重新生成导致的 token 级差异）
├── tools → 传入 toolUseContext.options.tools（父级完全相同的数组）
├── promptMessages → buildForkedMessages(prompt, assistantMessage)
│     克隆父级 assistant message 中所有 tool_use block
│     + 为每个 fork child 生成占位 tool_result
│     + 追加 per-child directive
└── forkContextMessages → 传入父会话完整消息历史
```

### 6.2 递归 Fork 防护

```
isInForkChild() 检查两个来源（OR 逻辑）：
  1. toolUseContext.options.querySource 包含 'agent:builtin:fork'
     （compaction 后仍有效——querySource 在 spawn 时设置，不被 compact 重写）
  2. toolUseContext.messages 中扫描到 fork 标记
     （fallback，防御 querySource 未传递的路径）

防护触发 → throw Error('Fork is not available inside a forked worker.')
```

## 七、Swarm — 多 Agent 编队

文件：`utils/swarm/` · `tasks/InProcessTeammateTask/`

### 7.1 Backend 体系

Swarm 抽象出两层接口：

```
PaneBackend（低层：pane 管理）
├── TmuxBackend         # tmux split-pane
├── ITermBackend        # iTerm2 native split-pane（需 it2 CLI）
└── InProcessBackend    # 同进程（无可视 pane）

TeammateExecutor（高层：Teammate 生命周期）
├── spawn(config)      → TeammateSpawnResult
├── sendMessage()      → 向 Teammate 邮箱写入
├── terminate()        → 优雅关闭（shutdownRequested flag）
└── kill()             → 强制终止

后端检测（backends/detection.ts）：
  自动探测环境 → 选择最优 backend：
  iTerm2 native > tmux > in-process
```

### 7.2 InProcessTeammateTask

文件：`tasks/InProcessTeammateTask/InProcessTeammateTask.tsx`

```
InProcessTeammateTaskState
├── agentId: "name@teamName"   # 格式化 ID，全局唯一
├── status: running | idle | completed | failed | killed
├── messages[]                 # 对话记录（上限 cap，防内存溢出）
├── pendingUserMessages[]      # Leader 注入消息队列
└── shutdownRequested          # 优雅关闭标志

生命周期：
  spawn → registerInProcessTeammateTask() → runInProcessRunner()
  idle  → createIdleNotification() → 通知 Leader
  kill  → killInProcessTeammate() → abort AbortController
```

**权限冒泡**（`createInProcessCanUseTool()`）：

```
Teammate 请求权限时：
  1. hasPermissionsToUseTool()  →  allow / deny → 直通
  2. behavior === 'ask' →
       优先：getLeaderToolUseConfirmQueue()  Leader UI 弹窗（共享主界面）
       fallback：mailbox 系统（写入 Leader 邮箱 → 轮询等待响应）
  3. Bash 命令 → 先尝试 ML 分类器自动批准（避免打扰 Leader）
```

### 7.3 Teammate 通信（Mailbox 系统）

```
writeToMailbox(agentId, message)   → 写入 Teammate 的收件箱
readMailbox(agentId)               → 读取未读消息
markMessageAsReadByIndex()         → 标记已读

消息类型：
  TEAMMATE_MESSAGE_TAG             # 普通 DM
  permission request / response    # 权限请求/应答
  idle notification                # Teammate 空闲通知
  shutdown request                 # 优雅停机指令
```

## 八、协调器模式（Coordinator Mode）

文件：`coordinator/coordinatorMode.ts`

**启用方式**：`CLAUDE_CODE_COORDINATOR_MODE=1`

### 8.1 模式特征

```
普通模式                           协调器模式
────────────────────────────────────────────────
自己执行工具                       专职编排，不直接干活
AgentTool 可选                    AgentTool 是主要工具
子 Agent 可同步                   所有子 Agent 强制异步
用户直接对话                       通过 task-notification 追踪进度
```

### 8.2 允许工具集

```
COORDINATOR_MODE_ALLOWED_TOOLS = [
  'Agent', 'SendMessage', 'TaskStop',   # 核心编排工具
  'TaskCreate', 'TaskGet', 'TaskList', 'TaskUpdate',  # 任务追踪
  'Read', 'Glob', 'Grep', 'WebSearch', 'WebFetch',    # 信息收集（只读）
  'AskUserQuestion', 'TodoWrite',       # 用户交互 + 待办列表
  'mcp__*'                             # MCP 工具
]
```

### 8.3 协调器系统提示关键内容

```
getCoordinatorSystemPrompt() 包含：
  ├── Phase 表格：Clarify → Plan → Spawn → Monitor → Synthesize
  ├── Worker 提示规范（如何构建高质量 prompt 传给子 Agent）
  ├── continue-vs-spawn 决策表（何时 SendMessage vs Agent）
  └── task-notification 格式说明（主 Agent 如何解读通知）
```

## 九、Agent 间消息工具

### 9.1 SendMessageTool

**用途**：向已命名的 Teammate 发送消息（协调器 → Worker DM）。

```
SendMessageTool.call({ to: agentName | 'all', message })
  ├── 找到 AppState.tasks 中 name === to 的活跃任务
  ├── writeToMailbox(taskId, { text: message, from: 'coordinator' })
  └── 返回确认或"agent not found"
```

### 9.2 TaskCreateTool / TaskGetTool / TaskListTool / TaskUpdateTool / TaskStopTool

这五个工具实现了**结构化任务追踪**，供协调器 Agent 管理工作单元：

```
TaskCreate  → 创建 TODO 条目（id, title, status, priority）
TaskUpdate  → 更新状态 / 描述（pending → in_progress → completed | failed）
TaskGet     → 读取单个任务详情
TaskList    → 列出所有任务（含过滤）
TaskStop    → 终止后台 Agent（kill AsyncAgentTask）
```

这套工具使协调器能维护**清晰的工作状态图**，而不是依赖线性对话历史。

## 十、数据流总结

```
用户输入
  │
  ▼
主 Agent query() 循环
  │  tool_use: Agent(...)
  ▼
AgentTool.call()
  │
  ├── Swarm？  →  spawnTeammate()
  │               └── InProcessBackend.spawn() / TmuxBackend.spawn()
  │                     └── runInProcessRunner() 或 tmux 命令
  │
  ├── Remote？ →  teleportToRemote()
  │               └── registerRemoteAgentTask()
  │                     └── pollRemoteSessionEvents() [后台轮询]
  │
  ├── Fork？   →  runAgent(FORK_AGENT, useExactTools=true)
  │               └── 继承父系统提示 + 工具 → 最大化 cache 命中
  │
  └── Normal → shouldRunAsync?
        │
        ├── YES → registerAsyncAgent()
        │           └── runAgent() [Promise，不阻塞]
        │               完成 → enqueuePendingNotification(task-notification)
        │
        └── NO  → await runAgent() [阻塞当前轮次]
                      完成 → 结果直接返回 tool_result

task-notification (user 角色) 注入主 Agent 消息队列
  → 主 Agent 下一轮感知到任务完成，继续后续流程
```

## 十一、关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 统一入口 | AgentTool 单点 | 避免多入口导致权限/日志/追踪分散 |
| 强制异步结果返回 | task-notification XML（user 角色注入） | 与主循环消息结构一致，不需要特殊 callback 机制 |
| Fork path cache 设计 | 完全相同的系统提示 + 工具数组 | Anthropic API prompt cache 以 token 级哈希为键，任何差异破坏命中 |
| worktree 清理策略 | 无修改时自动删除 | 节省磁盘，有修改时保留分支供 PR 使用 |
| Backend 抽象 | PaneBackend + TeammateExecutor 两层 | 低层 pane 管理与高层 Teammate 生命周期解耦 |
| 权限冒泡 | Leader UI 优先，mailbox 兜底 | Leader UI 提供最佳 UX（工具专属对话框）；mailbox 保证 headless 可用 |
| token 计数分离 | input: latest-only，output: cumulative | 避免 input tokens 在多轮对话中重复计算（prompt 在每轮都全量发送） |
| 递归 fork 防护 | querySource 检测（主）+ 消息扫描（兜底） | querySource 在 autocompact 后仍存活；消息扫描防 querySource 未注入的边缘路径 |
| 协调器模式工具限制 | 白名单（只读 + 编排工具） | 防止协调器"绕过分工"直接执行，保证架构清晰 |
