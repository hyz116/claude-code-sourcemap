# Claude Code 权限系统深度分析

## 1. 总览

权限系统是 Claude Code 的安全核心，负责决定每个工具调用是否被允许、拒绝或需要用户确认。系统设计上的几个关键考量：

- **分层决策**：规则 → 模式 → 交互，优先级严格有序
- **可绕过性受控**：`bypassPermissions` 不能绕过 safety check 和 content-specific ask 规则
- **多信号竞速**：交互时本地用户 / 远程 Bridge / Channel / Hook / Classifier 竞速，先到者胜
- **auto 模式 LLM 化**：用 LLM classifier 替代用户判断，有密度保护和 fail-close 机制

## 5. 核心决策管线（hasPermissionsToUseTool）

```
src/utils/permissions/permissions.ts
```

### 5.1 外层包装（hasPermissionsToUseTool）

在调用内层管线前，处理模式变换：

```
dontAsk 模式
  └→ 转为 bypassPermissions 处理

auto 模式（TRANSCRIPT_CLASSIFIER gate）
  ├→ acceptEdits 快速路径：文件编辑类工具直接 allow
  ├→ 允许工具列表（allowlisted）直接 allow
  └→ 调用 classifyYoloAction(messages, action, tools)
       ├→ 成功允许 → allow（重置连续拒绝计数）
       ├→ 成功拒绝 → 检查拒绝密度限制
       │    ├→ 超限（交互模式）→ 回退到手动提示
       │    └→ 超限（无头模式）→ throw AbortError
       ├→ transcriptTooLong → 回退手动 / AbortError（无头）
       └→ unavailable（API 错误）
            ├→ iron_gate_closed=true → fail-closed（deny + retry 提示）
            └→ iron_gate_closed=false → fail-open（回退手动）

shouldAvoidPermissionPrompts（无头/后台代理）
  ├→ 先运行 PermissionRequest hooks
  │    └→ hook 有决定 → 使用该决定
  └→ hook 无决定 → auto-deny（AUTO_REJECT_MESSAGE）
```

### 5.2 内层管线（hasPermissionsToUseToolInner）

17 个决策步骤，严格有序：

```
1a. 整个工具有 deny 规则？
    └→ deny（不可绕过）

1b. 整个工具有 ask 规则？
    ├→ 沙箱自动允许例外：BashTool + sandbox enabled + shouldUseSandbox(input)
    │    └→ 继续到 1c
    └→ ask（进入交互流程）

1c. tool.checkPermissions(parsedInput, context) 调用工具自身权限检查
    （Bash 在此检查子命令前缀规则、subcommandResults 等）

1d. checkPermissions 返回 deny？
    └→ deny

1e. tool.requiresUserInteraction() && toolPermissionResult 是 ask？
    └→ ask（bypass-immune：即使 bypassPermissions 也要问）

1f. checkPermissions 返回 ask + decisionReason.type='rule' + ruleBehavior='ask'？
    └→ ask（内容级 ask 规则，bypass-immune）

1g. checkPermissions 返回 ask + decisionReason.type='safetyCheck'？
    └→ ask（敏感路径保护，bypass-immune）

2a. bypassPermissions 模式？
    或（plan 模式 + isBypassPermissionsModeAvailable）？
    └→ allow（跳过所有后续步骤）

2b. 工具有 always-allow 规则？
    └→ allow

3.  passthrough → 转为 ask（准备进入用户交互）
```

### 5.3 仅规则检查（checkRuleBasedPermissions）

bypassPermissions 模式也需要检查的规则子集（步骤 1a-1g），不包含模式转换和分类器：

```typescript
export async function checkRuleBasedPermissions(
  tool, input, context
): Promise<PermissionAskDecision | PermissionDenyDecision | null>
```

### 7.3 recheckPermission 机制

用户交互期间若权限配置变更（如 CCR 切换模式），可触发 recheck：

```typescript
async recheckPermission() {
  const freshResult = await hasPermissionsToUseTool(tool, input, ...)
  if (freshResult.behavior === 'allow') {
    // claim() 原子检查，防止 in-flight 窗口冲突
    if (!claim()) return
    resolveOnce(ctx.buildAllow(...))
  }
}
```

|---------|---------|---------|

## 14. 设计决策

**1. safety check bypass-immune**
敏感路径（`.git/`, `.claude/`, shell 配置）不可绕过，防止 bypassPermissions 模式下意外破坏仓库或注入恶意配置。`classifierApprovable` 字段区分哪些 safety check 可以让 auto 模式 classifier 评估（有上下文可判断），哪些不行。

**2. iron_gate_closed fail-safe**
auto 模式 classifier 不可用时，通过 feature flag 控制 fail-closed（deny）vs fail-open（回退手动）。通过 `CLASSIFIER_FAIL_CLOSED_REFRESH_MS` 定期刷新，允许快速响应线上故障。

**3. 拒绝密度保护（Denial Limits）**
防止 auto 模式陷入持续拒绝死循环（防止 agent 卡死）和防止策略被反复规避（防止恶意使用）。异步子代理用 `localDenialTracking` 本地跟踪，避免 `setAppState` 的 no-op 问题。

**4. resolveOnce + claim() 原子模式**
5路竞速中，`claim()` 提供原子 check-and-mark，防止在异步 await 间隙（isResolved 检查到 resolve 调用之间）出现竞态条件。这是该系统最精密的并发设计。

**5. 宽容加载（Lenient Loading）**
写规则时读取现有规则，若 schema 验证失败（如 hooks 字段语法错误）不报错，而是宽容解析保留现有数据。这防止了一个字段的错误导致所有权限规则丢失。

**6. EDITABLE_SOURCES 限制**
policySettings 和 flagSettings 不可编辑，保证企业/托管环境的策略不被用户代码覆盖。代码中有运行时检查（`!EDITABLE_SOURCES.includes(rule.source)`）作为防御层。

**7. checkmark 留存时长**
Bash Classifier 自动允许后，checkmark 展示时长根据终端是否聚焦而不同（focused: 3s, unfocused: 1s），在"让用户看到结果"和"不阻塞下一个权限提示"之间平衡。
