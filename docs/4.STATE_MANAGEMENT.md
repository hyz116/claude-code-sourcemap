## 一、整体架构

Claude Code 的状态管理采用**自制极简 pub/sub Store + React `useSyncExternalStore`** 方案，
完全绕过 Redux/Zustand 等第三方库，在保持 React 声明式更新的同时支持工具层（非 React 代码）直接读写状态。

```
Store<AppState>                         (store.ts)
├── getState()    → 同步读取
├── setState()    → 不可变更新 + 通知订阅者
└── subscribe()   → 注册/注销监听器

        │ React Context
        ▼
AppStateProvider                        (AppState.tsx)
├── 持有唯一 Store 实例（useState 保证单例）
├── 暴露 AppStoreContext
├── useSettingsChange() 订阅磁盘配置变化
└── 嵌套 MailboxProvider / VoiceProvider

        │ useSyncExternalStore
        ▼
useAppState(selector)                   → 切片订阅，selector 不变则不重渲
useSetAppState()                        → 稳定 setState 引用，不触发重渲
useAppStateStore()                      → 完整 Store（getState + setState）
```

## 二、Store 原语

文件：`state/store.ts`

```typescript
type Store<T> = {
  getState: () => T
  setState: (updater: (prev: T) => T) => void
  subscribe: (listener: () => void) => () => void  // 返回取消订阅函数
}
```

**实现要点**：

| 机制 | 代码 | 目的 |
|------|------|------|
| 引用相等短路 | `if (Object.is(next, prev)) return` | 防止相同引用触发无效渲染 |
| 纯函数更新 | `updater: prev => next` | 不可变更新，React 18 兼容 |
| 同步 onChange hook | `onChange?.({ newState, oldState })` | Store 级别副作用（不经过 React） |
| listeners 用 Set | `new Set<Listener>()` | O(1) 添加/删除，遍历顺序稳定 |

`createStore` 接受可选的 `onChange` 回调，Claude Code 将 `onChangeAppState` 注入此处，实现状态变更驱动的外部同步（CCR / SDK / 配置文件）。

## 四、React 集成层

文件：`state/AppState.tsx`

### 4.1 Provider 树

```
AppStateProvider
  └── HasAppStateContext.Provider (value=true)   ← 嵌套检测
        └── AppStoreContext.Provider (value=store)
              └── MailboxProvider
                    └── VoiceProvider (ant-only / DCE)
                          └── {children}
```

**嵌套防护**：`HasAppStateContext` 检测到已在 Provider 内时 throw，防止意外双重 Store 实例。

### 4.2 Hooks API

```typescript
// 切片订阅（最常用）— 只在 selector 返回值变化时重渲
function useAppState<T>(selector: (state: AppState) => T): T
  // 实现：useSyncExternalStore(store.subscribe, () => selector(store.getState()))

// 获取 setState（稳定引用，不触发重渲）
function useSetAppState(): (updater: (prev: AppState) => AppState) => void

// 获取完整 Store（极少用于需要 getState 的场景）
function useAppStateStore(): AppStateStore

// 安全版本（Provider 外调用返回 undefined，不抛出）
function useAppStateMaybeOutsideOfProvider<T>(selector): T | undefined
```

**性能注意事项**（来自代码注释）：
- `selector` 不能返回新对象（`Object.is` 对象引用始终不等 → 无限重渲）
- 多个独立字段应多次调用 hook，而非返回对象：

```typescript
// ✅ 正确：各自订阅，互不影响
const verbose = useAppState(s => s.verbose)
const model = useAppState(s => s.mainLoopModel)

// ✅ 正确：返回已有子对象引用
const { text, promptId } = useAppState(s => s.promptSuggestion)

// ❌ 错误：每次渲染产生新对象引用 → 无限循环
const combo = useAppState(s => ({ verbose: s.verbose, model: s.mainLoopModel }))
```

### 4.3 与非 React 代码的集成

工具层代码（Tool.call, query.ts 等）通过 `ToolUseContext` 中的函数引用读写状态：

```
ToolUseContext
├── getAppState()   → () => store.getState()   [只读快照]
└── setAppState()   → store.setState           [直接写入，触发 pub/sub]
```

这使工具代码无需感知 React，同时变更仍能驱动 TUI 重渲染。

## 五、onChangeAppState — 状态变更副作用

文件：`state/onChangeAppState.ts`

作为 `createStore` 的 `onChange` 回调，每次 `setState` 后同步执行 diff，驱动外部系统同步：

```
监听字段                  → 副作用

toolPermissionContext.mode → notifySessionMetadataChanged()  (CCR external_metadata 同步)
                          → notifyPermissionModeChanged()    (SDK status stream)
                          注：internal-only mode (bubble, ungated-auto) 不传播到 CCR

mainLoopModel             → updateSettingsForSource('userSettings', { model })
                          → setMainLoopModelOverride()

expandedView              → saveGlobalConfig({ showExpandedTodos, showSpinnerTree })

verbose                   → saveGlobalConfig({ verbose })

tungstenPanelVisible      → saveGlobalConfig({ tungstenPanelVisible })  (ant-only)

settings                  → clearApiKeyHelperCache()
                          → clearAwsCredentialsCache() / clearGcpCredentialsCache()
                          → settings.env 变化时 applyConfigEnvironmentVariables()
```

**设计意图**：所有向外同步的逻辑集中在这一处，避免在 8+ 个独立 `setAppState` 调用点中各自散布通知代码，防止遗漏。

## 六、Task 状态框架

文件：`utils/task/framework.ts`

Task 状态是 AppState 中最动态的部分，由 framework.ts 提供通用原语：

### 6.1 核心操作

```typescript
// 类型安全的原子更新（返回同引用则跳过 spread，防止无效渲染）
function updateTaskState<T extends TaskState>(
  taskId: string,
  setAppState: SetAppState,
  updater: (task: T) => T,
): void

// 注册新任务 + 发射 SDK task_started 事件
// resume 时携带已有 UI 状态（retain, messages, diskLoaded）
function registerTask(task: TaskState, setAppState: SetAppState): void

// 主动驱逐：任务 terminal + notified + 超过 grace period → 从 Map 删除
function evictTerminalTask(taskId, setAppState): void
```

### 6.2 磁盘轮询（运行中任务）

后台任务的输出写入磁盘文件，framework 轮询增量内容：

```
pollTasks()  [每 1000ms 调用]
  │
  ├── generateTaskAttachments()
  │     ├── running 任务 → getTaskOutputDelta(id, outputOffset) [读磁盘]
  │     └── terminal + notified → 列为 evictedTaskIds
  │
  ├── applyTaskOffsetsAndEvictions()   [TOCTOU 安全：基于 fresh prev.tasks]
  │     ├── 更新 running 任务的 outputOffset
  │     └── 删除 evictedTaskIds（re-check terminal + notified + grace period）
  │
  └── enqueueTaskNotification()  [任务完成时推送 task-notification XML]
```

**TOCTOU 保护**：`generateTaskAttachments` 是 async 的（磁盘 IO），在 await 期间任务状态可能改变（如 completed）。`applyTaskOffsetsAndEvictions` 基于 `fresh prev.tasks` 重新检查，不使用 await 前的快照，防止用过期状态覆盖已完成任务。

### 6.3 Task 生命周期常数

| 常数 | 值 | 用途 |
|------|---|------|
| `POLL_INTERVAL_MS` | 1000ms | 磁盘轮询频率 |
| `STOPPED_DISPLAY_MS` | 3s | killed 任务消失前的展示时间 |
| `PANEL_GRACE_MS` | 30s | 协调器面板 terminal 任务的宽限期（用户阅读） |

## 八、Teammate 视图辅助

文件：`state/teammateViewHelpers.ts`

管理"查看 Agent 对话记录"时的内存与 UI 状态切换：

```
enterTeammateView(taskId)
  ├── retain: true       → 阻止 local_agent 任务被驱逐
  ├── evictAfter: undefined → 清除已有驱逐计时器
  ├── viewingAgentTaskId → taskId
  ├── viewSelectionMode  → 'viewing-agent'
  └── 如果之前在查看另一个 Agent → release() 它（释放内存，重新启动计时器）

exitTeammateView()
  ├── 对 local_agent: retain: false, messages: undefined, diskLoaded: false
  ├── 如果已 terminal: evictAfter = now + PANEL_GRACE_MS（行保留 30s）
  └── viewingAgentTaskId → undefined

stopOrDismissAgent(taskId)
  ├── running → abortController.abort()（中止执行）
  └── terminal → evictAfter: 0（立即隐藏行）
```

**内存策略**：local_agent 任务的 `messages[]` 默认不保留（stub 形式），只有当用户主动查看对话记录时才加载（retain: true + 从磁盘读取），退出时释放。这避免了大量后台任务堆积 Message 数组占用内存。

## 九、DeepImmutable 类型策略

```typescript
export type AppState = DeepImmutable<{
  settings: SettingsJson
  verbose: boolean
  // ... 大多数字段
}> & {
  // 排除在 DeepImmutable 之外的字段（含函数类型）
  tasks: { [taskId: string]: TaskState }        // TaskState 含 abortController 等函数
  agentNameRegistry: Map<string, AgentId>        // Map 不兼容 DeepImmutable
  // ... 其他含特殊类型的字段
}
```

`DeepImmutable` 使 TypeScript 在编译期拒绝直接变异，强制通过 `setState(prev => ...)` 进行不可变更新，但对含函数/Map/Set 等原生类型的字段必须手动排除。

## 十、关键设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 自制 Store | 35 行 createStore | 无第三方依赖；与 React 18 useSyncExternalStore 直接对接；全控行为 |
| Object.is 短路 | setState 入口检查 | 工具层代码会频繁 setAppState；相同引用跳过通知，防止无效渲染级联 |
| onChange hook | onChangeAppState 集中注册 | 避免 8+ 个 setAppState 调用点各自维护外部同步，防止遗漏 |
| tasks 排除 DeepImmutable | TaskState 含函数类型 | TS 的 DeepImmutable 遇到函数会变成 never；手动维护不可变约定 |
| Task 不在 onChange 处理完成通知 | 各 Task 类型自己 enqueue | 避免 onChange 与 Task 回调竞争导致双发；onChange 同步执行，Task 回调异步 |
| TOCTOU 安全写回 | applyTaskOffsetsAndEvictions 基于 fresh state | generateTaskAttachments 含磁盘 await，期间 task 可能已完成，必须在回调里重检查 |
| Teammate 消息懒加载 | retain 标志控制内存 | 后台可能有数十个 Agent，全量保留 messages 会撑爆内存；只在用户查看时加载 |
| selector 必须返回现有引用 | useAppState 文档约束 | React Compiler + useSyncExternalStore 依赖 Object.is；新对象导致每帧重渲 |
